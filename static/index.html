What is ReAct?

ReAct = Reasoning + Acting
A loop where the LLM alternates between thinking (what do I still need to know?) and acting (call a tool to find out) — repeating until it has enough information to write a final answer. Each round the LLM sees all previous messages + tool results, so its decisions get progressively better informed.

Concrete Example — CMP-4491 "Not Delivering"
Round LLM calls these tools What it learns
1 getCampaignStatus searchKnowledgeBase status=PAUSED, budget=$0.00, bid=$0.95 · KB returns delivery checklist (doc-001)
2 getHourlySpendCurve getPacingSettings Budget hit at 08:03 AM · pacingMode=ASAP (root cause confirmed)
3 — no tool calls — Synthesises final answer: 4 issues found, fix recommendations → RESOLVED
Why parallel tool execution matters: In Round 1, getCampaignStatus and searchKnowledgeBase run at the same time via LangGraph's ToolNode — not sequentially. This cuts latency significantly when the LLM requests multiple tools at once.

How thread_id links turns together
graph.py — add checkpointer to compile() from langgraph.checkpoint.memory import MemorySaver compiled = graph.compile(checkpointer=MemorySaver()) routes.py — pass thread_id per session result = await agent.ainvoke( {"messages": [HumanMessage(content=user_text)]}, config={"configurable": {"thread_id": body.advertiser_id}} ) thread_id options: advertiser_id → one shared memory per advertiser (across all their campaigns) session_id → one memory per browser tab / API session ticket_id → one memory per support conversation thread
Enabled: Multi-turn memory is active in this project. Toggle the Multi-turn Memory switch in New Query to turn it on. When ON, the same thread_id is sent with every query — agent accumulates history across clicks. When OFF, a unique thread_id is generated per request (fresh state every time).

What gets saved per thread_id
messages full history — HumanMessage, AIMessage, ToolMessage across all turns versions_seen which checkpoint version each channel last saw (conflict detection) pending_sends queued messages for human interrupt resume (empty when no interrupt) step total node executions so far in this thread
Production recommendation: Use AsyncPostgresSaver pointing at the same Cloud SQL instance already used for tickets. LangGraph creates its own tables (checkpoints, checkpoint_blobs, checkpoint_writes) — no conflict with the tickets table. MemorySaver on Cloud Run with 2+ instances means each instance has separate RAM — same thread_id routed to different instances loses history.
